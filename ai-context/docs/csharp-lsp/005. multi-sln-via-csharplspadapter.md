# 多 sln 注入：基于 CSharpLspAdapter 扩展

> **背景**：[`multi-sln-router-impl.md`](./004. multi-sln-router-impl.md) §2 方案 A 是从零写一个 TS 中间层（`inject-multi-sln.ts`），让它在 Claude Code 和 Roslyn LSP 之间做 stdio 代理。  
> **本文思路**：[`agasper/CSharpLspAdapter`](https://github.com/agasper/CSharpLspAdapter) 已经是个生产级 LSP 代理框架，源码就在 [`external/CSharpLspAdapter`](../../../external/CSharpLspAdapter)。**fork 它 + ~30 行 C# 增量即可实现多 sln 注入**，比从零搭脚手架优雅 5 倍。

## 1. CSharpLspAdapter 现有架构（关键点）

### 1.1 入口：spawn 下游 + 双向异步循环

```@e:/claude-hub/claude-code/external/CSharpLspAdapter/Program.cs:23-50
    public static async Task<int> Main(string[] args)
    {
        try
        {
            var csharpLsPath = FindCSharpLs();
            if (csharpLsPath == null) { /* ... */ return 1; }

            _serverProcess = StartServer(csharpLsPath, args);

            using var cts = new CancellationTokenSource();

            var clientToServer = ProcessClientMessages(_serverProcess, cts.Token);
            var serverToClient = ProcessServerMessages(_serverProcess, cts.Token);

            await Task.WhenAny(clientToServer, serverToClient).ConfigureAwait(false);
```

- Client 端（stdin/stdout）= LSP 客户端（Claude Code）
- Server 端 = 子进程 csharp-ls 或 roslyn-language-server
- 两个独立异步循环：`ProcessClientMessages`（C→S）和 `ProcessServerMessages`（S→C）

### 1.2 Client→Server 路径已有"改写钩子"

```@e:/claude-hub/claude-code/external/CSharpLspAdapter/Program.cs:147-172
    private static async Task ProcessClientMessages(Process server, CancellationToken ct)
    {
        LogDebug("ProcessClientMessages started");
        await using var reader = Console.OpenStandardInput();
        await using var writer = server.StandardInput.BaseStream;

        while (!ct.IsCancellationRequested && !server.HasExited)
        {
            var message = await ReadLspMessage(reader, "client", ct).ConfigureAwait(false);
            if (message == null) break;

            // Transform client messages if needed (e.g., enable capabilities)
            message = TransformClientMessage(message);

            // Forward to server (with lock since ProcessServerMessages also writes to server)
            await WriteLspMessageWithLock(writer, message, ServerWriteLock, ct).ConfigureAwait(false);
        }
    }
```

`TransformClientMessage` 当前做两件事：
- 检测 `initialize` 请求 → 强制 `capabilities.workspace.configuration = true`
- 规范化所有 file:// URI（修 Windows 双斜杠问题）

**多 sln 注入只需要在 `WriteLspMessageWithLock` 之后插一个钩子：检测刚转发的是不是 `initialized` 通知，是则批量发 `solution/open`。**

## 2. 多 sln 注入的最小 patch

### 2.1 触发点选择

| 选项 | 时序 | 是否可用 |
|------|------|--------|
| `initialize` 响应到达时 | server 刚响应能力协商，**还没准备好接收项目加载** | ❌ 太早 |
| `initialized` 通知转发后 | LSP 握手完成；spec 规定之后才允许文件操作 | ✅ **最佳** |
| `textDocument/didOpen` 时 | 太晚，已经会触发 fallback `--autoLoadProjects` | ❌ |

为什么 Roslyn LSP 接受连续多次 `solution/open`：

```@e:/claude-hub/claude-code/external/roslyn/src/LanguageServer/Microsoft.CodeAnalysis.LanguageServer/HostWorkspace/LanguageServerProjectSystem.cs:55-67
    public async Task OpenSolutionAsync(string solutionFilePath)
    {
        _logger.LogInformation(string.Format(LanguageServerResources.Loading_0, solutionFilePath));
        _hostProjectFactory.SolutionPath = solutionFilePath;

        var (_, projects) = await SolutionFileReader.ReadSolutionFileAsync(solutionFilePath, DiagnosticReportingMode.Throw, CancellationToken.None);
        foreach (var (path, guid) in projects)
        {
            await BeginLoadingProjectAsync(path, guid);
        }
        await WaitForProjectsToFinishLoadingAsync();
        await ProjectInitializationHandler.SendProjectInitializationCompleteNotificationAsync();
    }
```

**追加语义**——每次调用都把 sln 里的 csproj 加到同一个 workspace，不卸载之前的。所以 N 次 `solution/open` = 合并 N 个 sln 的 csproj。

### 2.2 完整 C# patch（基于 v1.0.2）

#### 新增字段（顶部）

```csharp
// 复数（新）：逗号分隔多个 sln 路径
// 兼容单数（旧）：LSP_SOLUTION_PATH 仍生效（用作 workspace/configuration 应答）
private static readonly string[] SolutionPaths = ParseSolutionPaths();

private static string[] ParseSolutionPaths()
{
    var raw = Environment.GetEnvironmentVariable("LSP_SOLUTION_PATHS")
              ?? Environment.GetEnvironmentVariable("LSP_SOLUTION_PATH")
              ?? "";
    return raw
        .Split(new[] { ',' }, StringSplitOptions.RemoveEmptyEntries)
        .Select(p => p.Trim())
        .Where(p => p.Length > 0)
        .ToArray();
}
```

#### 修改 `ProcessClientMessages`（加 ~15 行）

```csharp
private static async Task ProcessClientMessages(Process server, CancellationToken ct)
{
    LogDebug("ProcessClientMessages started");
    await using var reader = Console.OpenStandardInput();
    await using var writer = server.StandardInput.BaseStream;

    var injected = false;   // 局部状态：注入只执行一次

    while (!ct.IsCancellationRequested && !server.HasExited)
    {
        var message = await ReadLspMessage(reader, "client", ct).ConfigureAwait(false);
        if (message == null) break;

        message = TransformClientMessage(message);
        await WriteLspMessageWithLock(writer, message, ServerWriteLock, ct).ConfigureAwait(false);

        // === NEW: 在 initialized 转发后注入多个 solution/open ===
        if (!injected && SolutionPaths.Length > 0 && IsNotification(message, "initialized"))
        {
            injected = true;
            await InjectSolutionOpenAsync(writer, ct).ConfigureAwait(false);
        }
    }
    LogDebug("ProcessClientMessages ended");
}

private static bool IsNotification(string message, string expectedMethod)
{
    try
    {
        var json = JsonNode.Parse(message);
        return json?["method"]?.GetValue<string>() == expectedMethod
            && json?["id"] == null;     // 通知没有 id
    }
    catch { return false; }
}

private static async Task InjectSolutionOpenAsync(Stream writer, CancellationToken ct)
{
    foreach (var slnPath in SolutionPaths)
    {
        try
        {
            var fullPath = Path.GetFullPath(slnPath);
            if (!File.Exists(fullPath))
            {
                LogDebug($"[multi-sln] WARN: file not found, skipping: {fullPath}");
                continue;
            }
            var uri = new Uri(fullPath).AbsoluteUri;
            var notif = $$"""
                {"jsonrpc":"2.0","method":"solution/open","params":{"solution":"{{uri}}"}}
                """;
            await WriteLspMessageWithLock(writer, notif, ServerWriteLock, ct).ConfigureAwait(false);
            LogDebug($"[multi-sln] Injected solution/open: {uri}");
        }
        catch (Exception ex)
        {
            LogDebug($"[multi-sln] ERROR injecting {slnPath}: {ex.Message}");
        }
    }
}
```

### 2.3 改 `FindCSharpLs()` 兼容 Roslyn LSP

private static string? FindCSharpLs()
{
    // 0. 显式环境变量优先
    var explicitPath = Environment.GetEnvironmentVariable("LSP_SERVER_PATH");
    if (!string.IsNullOrEmpty(explicitPath) && File.Exists(explicitPath))
        return explicitPath;

    // 1. 候选名（Windows 加 .cmd）
    var binNames = OperatingSystem.IsWindows()
        ? new[] { "csharp-ls.cmd", "csharp-ls.exe", "roslyn-language-server.cmd", "roslyn-language-server.exe" }
        : new[] { "csharp-ls", "roslyn-language-server" };

    var toolsDir = Path.Combine(Environment.GetFolderPath(Environment.SpecialFolder.UserProfile), ".dotnet", "tools");

    foreach (var name in binNames)
    {
        var path = Path.Combine(toolsDir, name);
        if (File.Exists(path)) return path;
    }

    // 2. fallback: PATH 搜索（原逻辑）
    // ...
}

## 6. 已知限制

### 6.1 仍是单 Roslyn LSP 进程

合并加载策略继承方案 A：所有 sln 的 csproj 并集都在一个 `Workspace` 里。如果总和 > 4-8 GB，需要走多进程路由（[`multi-sln-router-impl.md`](./004. multi-sln-router-impl.md) §3 方案 B），那个就不是 fork CSharpLspAdapter 能搞定的（要写多 server 协调 + 广播聚合，工作量量级变大）。

### 6.2 启动期 didOpen 时序

若 Claude Code 在 `initialized` 之后、`solution/open` 加载完成之前就发了 `textDocument/didOpen`，Roslyn 内部会把这些 didOpen queue 起来等加载完成。**通常无害**，但首次启动期文件操作的延迟会比稳态高。把 `startupTimeout` 调到 10 分钟。

### 6.3 不要在 `solution/open` 后 explicitly 发 `initialized`

`initialized` 是客户端→服务器，且 LSP spec 规定整个会话只发一次。我们的 patch 是**在客户端的 `initialized` 转发后**追加 `solution/open`，没重发 `initialized`，符合协议。

### 6.4 与 `--autoLoadProjects` 的关系

下游 Roslyn LSP 启动参数最好**不带** `--autoLoadProjects`（避免它再扫一遍）。但即使带了也无害——`solution/open` 的追加语义会和 `--autoLoadProjects` 的加载合并，去重在 Roslyn 内部按 csproj 绝对路径完成。

推荐：下游 server 启动参数只留 `--stdio --logLevel Information`：

```jsonc
// 在 plugin.json 里把额外参数传给 adapter（adapter 再转给 server）
{
  "command": "multi-sln-lsp-adapter",
  "args": ["--stdio", "--logLevel", "Information"],   // 这些会被透传
  ...
}
```
