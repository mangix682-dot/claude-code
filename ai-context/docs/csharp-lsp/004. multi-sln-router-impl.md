# 多 sln C# LSP 中间层路由：设计与实施方案

> **场景**：中型游戏 mono-repo 根目录下 3-4 个 sln（如 `Server.sln`、`Client.sln`、`Combat.sln`、`Tools.sln`），多个 sln 之间 csproj **可能重叠**（典型如 `Shared/*.csproj` 被几个 sln 同时引用）。  
> **目标**：写一个中间层让 Claude Code 的 AI **能够跨 sln 找符号、定义、引用**，又**不爆内存**、不必把所有 sln 永远全部加载。  
> **输出**：可落地的中间层（LSP Proxy / Router）设计、代码骨架、Claude Code 接入清单。  
> **基础读物**：先读完下列任意一篇，再读本文档收益最大。
> - [`dotrush-multi-sln.md`](./dotrush-multi-sln.md) — DotRush 多 sln 加载原理
> - [`plan-b-local-marketplace.md`](./002. plan-b-local-marketplace.md) §5.1 — Roslyn LSP 项目加载机制

---

## 0. TL;DR：三个方案速选

| 方案 | 实现成本 | 内存预算 | 跨 sln 能力 | 推荐使用场景 |
|------|---------|---------|------------|------------|
| **A. 单进程 + 多 `solution/open` 注入** | ★（半天） | **高**（单进程合并所有 csproj） | 自然全局 | 全部 sln 加起来内存 < 4 GB 可接受 |
| **B. 多进程路由（Router）** | ★★★（1 周） | **可控**（按需启停每个 sln 进程） | 路由 + 聚合 | 内存有限 / 希望物理隔离 / 经常只在一个 sln 工作 |
| **C. DotRush 作为后端** | ★★（2 天） | 中（DotRush 自管） | **内置支持** | 接受 DotRush 协议兼容性需要验证 |

**首选推荐**：先用 **方案 A** 走通端到端（实现最简单，1 个文件 ~100 行 TS），如果内存吃不下再切 **方案 B**；**方案 C** 作为对比基线，永远值得跑一遍验证 DotRush 在你工程下的表现。

> 🎯 **方案 A 的优化路径**：与其从零写 TS 中间层，可以直接 **fork [agasper/CSharpLspAdapter](https://github.com/agasper/CSharpLspAdapter)**（源码在 [`external/CSharpLspAdapter`](../../../external/CSharpLspAdapter)）—— 它已经是个生产级 LSP 代理框架，**~30 行 C# 增量就能加多 sln 注入**，且**自带** LSP framing、URI 规范化、`client/registerCapability` / `workspace/configuration` / `window/workDoneProgress/create` 拦截。**决定走方案 A 前强烈推荐先看**：[`multi-sln-via-csharplspadapter.md`](./005. multi-sln-via-csharplspadapter.md)。

## 1. 场景与约束分析

### 1.1 仓库结构假设

```
GameRepo/
├── Server.sln              # Server/*.csproj + Shared/*.csproj
├── Client.sln              # Client/*.csproj + Shared/*.csproj + Combat/Runtime.csproj
├── Combat.sln              # Combat/*.csproj + Shared/Common.csproj
├── Tools.sln               # Tools/*.csproj + Shared/Common.csproj
├── Server/                 # 60 csproj
├── Client/                 # 80 csproj
├── Combat/                 # 40 csproj
├── Tools/                  # 20 csproj
└── Shared/                 # 30 csproj  ← 被多个 sln 引用
```

### 1.2 csproj 重叠分析

重叠 csproj **不是冲突**——Roslyn `Workspace` 用 csproj **绝对路径**作为 `ProjectId` 的去重键。

> 源码证据：DotRush 通过 `MSBuildWorkspace.OpenProjectAsync(path)` 反复追加同一路径时，Roslyn 内部用路径相等性判定，不会真的加两次。  
> 见 `@e:/claude-hub/claude-code/external/DotRush/src/DotRush.Roslyn.Workspaces/ProjectsController.cs:26-53` 与 Roslyn `MSBuildWorkspace` 公开 API 行为。

所以：
- **单 workspace 合并所有 sln**：`Shared/Common.csproj` 只占一份内存，跨 sln 的 ProjectReference 自然解析
- **多 workspace 隔离**：`Shared/Common.csproj` 在每个独立进程里都会被加载一次（N 倍内存），跨 sln 引用需要 router 自己做聚合

### 1.3 AI 的典型查询模式

| 操作 | LSP 方法 | 是否跨 sln | 路由策略 |
|------|---------|----------|---------|
| 跳定义 | `textDocument/definition` | 通常单 sln（点哪查哪） | **按文件归属路由到唯一实例** |
| 找引用 | `textDocument/references` | **跨 sln**（Shared 的方法可能被四处调） | **广播 + 合并去重** |
| 找实现 | `textDocument/implementation` | 跨 sln | 广播 + 合并去重 |
| 工作区符号搜索 | `workspace/symbol` | **必须跨 sln**（AI "找 PlayerManager"） | **广播 + 合并** |
| Hover / 文档符号 | `textDocument/hover` / `documentSymbol` | 单 sln | 按文件路由 |
| 调用层级 | `callHierarchy/*` | 跨 sln | 广播 + 合并 |

**结论**：路由策略必须**按 LSP 方法分类**——一刀切（永远广播 or 永远路由单点）都不对。

## 2. 方案 A：单进程 + 多 `solution/open` 注入

### 2.1 原理

利用 Roslyn LSP 的两个关键事实：

1. **Workspace 是单例**（不管来几次 `solution/open` 都加到同一个）
2. **`solution/open` 是追加语义**——`LanguageServerProjectSystem.OpenSolutionAsync` 把 sln 里的 csproj 全部 `BeginLoadingProjectAsync`

   ```@e:/claude-hub/claude-code/external/roslyn/src/LanguageServer/Microsoft.CodeAnalysis.LanguageServer/HostWorkspace/LanguageServerProjectSystem.cs:55-67
       public async Task OpenSolutionAsync(string solutionFilePath)
       {
           _logger.LogInformation(string.Format(LanguageServerResources.Loading_0, solutionFilePath));
           _hostProjectFactory.SolutionPath = solutionFilePath;

           var (_, projects) = await SolutionFileReader.ReadSolutionFileAsync(solutionFilePath, DiagnosticReportingMode.Throw, CancellationToken.None);
           foreach (var (path, guid) in projects)
           {
               await BeginLoadingProjectAsync(path, guid);
           }
           await WaitForProjectsToFinishLoadingAsync();
           ...
       }
   ```

3. **官方 LSP 客户端（VSCode）只发一次** `solution/open`——所以不是协议层的限制，是客户端没写

中间层只需要做一件事：**初始化握手后，主动注入 N 次 `solution/open`**，其余消息透传。

### 2.2 架构

```
┌──────────────┐   stdio   ┌─────────────┐  stdio  ┌────────────────────┐
│ Claude Code  │ ───────▶  │ inject.ts   │ ──────▶ │ Roslyn LSP (单进程) │
│ LSP Client   │           │ (本方案)    │         │ workspace 合并 sln  │
└──────────────┘ ◀─────── └─────────────┘ ◀────── └────────────────────┘
                  双向透传 + 启动后注入 N 个 solution/open 通知
```

### 2.3 代码骨架（TypeScript / Node / Bun）

`scripts/inject-multi-sln.ts`（独立可运行，约 100 行）：

```typescript
#!/usr/bin/env bun
import { spawn } from 'node:child_process'
import {
  createMessageConnection,
  StreamMessageReader,
  StreamMessageWriter,
  type MessageConnection,
} from 'vscode-jsonrpc/node'
import { pathToFileURL } from 'node:url'

// ─── 配置 ────────────────────────────────────────────────────────
const ROSLYN_BIN = process.env.ROSLYN_LSP_BIN ?? 'roslyn-language-server'
const SLN_PATHS = (process.env.SLN_PATHS ?? '')
  .split(',')
  .map(s => s.trim())
  .filter(Boolean)
const ROSLYN_ARGS = process.argv.slice(2) // 透传给 Roslyn LSP

if (SLN_PATHS.length === 0) {
  console.error('[inject-multi-sln] SLN_PATHS env required, e.g. "C:/repo/A.sln,C:/repo/B.sln"')
  process.exit(1)
}

// ─── 启动 Roslyn LSP 子进程 ───────────────────────────────────────
const child = spawn(ROSLYN_BIN, ROSLYN_ARGS, {
  stdio: ['pipe', 'pipe', 'inherit'],
})

// ─── 建立两条 LSP 连接：上游(Claude Code) / 下游(Roslyn) ──────────
const upstream: MessageConnection = createMessageConnection(
  new StreamMessageReader(process.stdin),
  new StreamMessageWriter(process.stdout),
)
const downstream: MessageConnection = createMessageConnection(
  new StreamMessageReader(child.stdout!),
  new StreamMessageWriter(child.stdin!),
)

// ─── 双向透传：所有请求/通知双向 forward ──────────────────────────
function pipe(from: MessageConnection, to: MessageConnection) {
  from.onRequest(async (method, params) => to.sendRequest(method, params))
  from.onNotification((method, params) => to.sendNotification(method, params))
}
pipe(upstream, downstream)
pipe(downstream, upstream)

// ─── 关键：拦截 'initialized' 通知后，注入多个 solution/open ──────
let injected = false
upstream.onNotification('initialized', async params => {
  await downstream.sendNotification('initialized', params)
  if (injected) return
  injected = true

  for (const sln of SLN_PATHS) {
    const uri = pathToFileURL(sln).toString()
    console.error(`[inject-multi-sln] solution/open -> ${uri}`)
    await downstream.sendNotification('solution/open', { solution: uri })
  }
})

upstream.listen()
downstream.listen()

child.on('exit', code => process.exit(code ?? 0))
```

> ⚠️ **注意**：`vscode-jsonrpc` 的 `pipe()` 双向透传写法略简化，实际生产实现需要处理 `onRequest` 双注册冲突（在同一 connection 上 `initialized` 既透传又拦截）。最稳的做法是用低层 `Message` 处理器 + 手动 framing。仓库里已经有用 `vscode-jsonrpc` 的成熟样例：[`@e:/claude-hub/claude-code/src/services/lsp/LSPClient.ts:1-50`](e:/claude-hub/claude-code/src/services/lsp/LSPClient.ts)

### 2.4 优劣

✅ **优点**
- 实现极简（~100 行 TS，半天搞定）
- 跨 sln 引用查找**完全免费**——所有 csproj 都在同一 `MSBuildWorkspace.CurrentSolution`
- 重叠 csproj **自动去重**（Roslyn `ProjectId` 路径键）

❌ **缺点**
- 内存上限 = **所有 sln 中 csproj 的并集**；大型工程可能 4-8 GB
- 启动慢（一次性 RestoreProjects + LoadProjects 上百个 csproj，分钟级）
- 出问题就是整体崩溃，没有进程隔离

### 2.5 适配 Claude Code

在你的本地 marketplace 插件（参考 [`plan-b-local-marketplace.md`](./002. plan-b-local-marketplace.md)）的 `plugin.json` 里把命令换成 inject.ts：


## 3. 方案 B：多进程路由 Router（推荐用于内存受限）

### 3.1 架构图

```
                          ┌──────────────────────────────┐
                          │  Router (本方案 ~ 400 行 TS) │
┌──────────────┐  stdio   │  ┌────────────────────────┐  │
│ Claude Code  │ ───────▶ │  │ 路由表 file→[sln*]     │  │
│ LSP Client   │          │  │ 容量管理 / 健康检查    │  │
└──────────────┘ ◀─────── │  └────────────────────────┘  │
                          └─────┬───────┬───────┬────────┘
                                │       │       │
                       spawn lazy/eager │       │
                                │       │       │
                       ┌────────▼─┐ ┌───▼────┐ ┌▼────────┐
                       │Roslyn LSP│ │Roslyn  │ │Roslyn   │
                       │--solution│ │ Client │ │ Combat  │
                       │  Server  │ │  .sln  │ │  .sln   │
                       └──────────┘ └────────┘ └─────────┘
```

### 3.2 进程模型

| 项 | 选择 |
|----|------|
| 每个 sln 一个 Roslyn LSP | ✅ |
| Roslyn 启动参数 | `--solutionPath <sln>`（**限定范围、避免 autoLoadProjects OOM**） |
| 启动时机 | **懒启动**（首次访问该 sln 的文件时启动） |
| 闲置回收 | 可选，30 min 无活动后 stop（释放内存） |
| 健康检查 | 每分钟 ping `workspace/symbol`，崩溃自动 respawn |
| 总进程数 | 等于活跃 sln 数（典型 1-2 个） |

### 3.3 路由表数据结构

```typescript
type Router = {
  instances: Map<string, RoslynInstance>          // slnName -> 实例
  fileToSlnNames: Map<string, string[]>           // csproj 绝对路径 -> 包含它的 sln 列表
  slnToFiles: Map<string, Set<string>>            // sln -> 它包含的 csproj 路径集合
  capabilitiesIntersection: ServerCapabilities    // 所有实例 capability 的交集
}
```

**`fileToSlnNames` 构建**：启动时用 [`MSBuildSolutionLoader`](./dotrush-multi-sln.md#4-sln-文件解析msbuildsolutionloader) 同款思路（`Project(...)` 行扫描）解析每个 sln，得到 `sln -> [csproj...]` 映射，反向得到 `csproj -> [sln...]`。

### 3.4 LSP 消息分类与路由策略

| 阶段 / 方法 | 策略 | 备注 |
|-----------|------|-----|
| `initialize` 请求 | 不广播；只返回 router 自己的能力（capabilities 取交集） | 子实例的 initialize 在首次 spawn 时延迟做 |
| `initialized` 通知 | 透传到子实例（spawn 后）的初始化流程 | |
| `textDocument/didOpen` | 解析文件所在 csproj → 路由到包含该 csproj 的所有 sln 实例 | 重叠 csproj 多份 didOpen |
| `textDocument/didChange` `didSave` `didClose` | 同上 | 状态同步必须一致 |
| `textDocument/definition` `hover` `documentSymbol` `signatureHelp` | **任选一个**包含该文件的实例发请求，返回单一结果 | 重叠 csproj 取第一个就行（语义一致） |
| `textDocument/references` `implementation` `callHierarchy/*` | **广播**到所有已启动的实例 → 合并 + 按 `(uri, range)` 去重 | 跨 sln 必须 |
| `workspace/symbol` | **广播**到所有已启动 + 配置上"按需 spawn"开关 | AI 找符号场景关键 |
| `workspace/configuration` | router 自己应答（参考 `@e:/claude-hub/claude-code/src/services/lsp/manager.ts` 已有实现） | |
| `client/registerCapability` `window/workDoneProgress/create` | router 应答 `null` | Claude Code LSP 客户端没实现 |

```typescript
  createMessageConnection,
  StreamMessageReader,
  StreamMessageWriter,
  type MessageConnection,
} from 'vscode-jsonrpc/node'
import { RoslynInstance } from './instance.js'
import { parseSolutionCsprojs } from './sln-parser.js'
import type { RouterConfig } from './config.js'

export async function startRouter(cfg: RouterConfig) {
  // 1. 构建文件→sln 映射
  const fileToSln = new Map<string, string[]>()
  const slnToFiles = new Map<string, Set<string>>()
  for (const [name, slnPath] of Object.entries(cfg.slns)) {
    const csprojs = await parseSolutionCsprojs(slnPath)
    slnToFiles.set(name, new Set(csprojs))
    for (const cs of csprojs) {
      const list = fileToSln.get(cs) ?? []
      list.push(name)
      fileToSln.set(cs, list)
    }
  }

  // 2. 准备实例容器（不立即启动）
  const instances = new Map<string, RoslynInstance>()
  for (const [name, slnPath] of Object.entries(cfg.slns)) {
    instances.set(name, new RoslynInstance(name, slnPath, cfg))
  }

  // 3. 上游与 Claude Code 通信
  const upstream = createMessageConnection(
    new StreamMessageReader(process.stdin),
    new StreamMessageWriter(process.stdout),
  )

  // —— initialize：返回交集 capabilities ——
  upstream.onRequest('initialize', async params => {
    return {
      capabilities: cfg.aggregatedCapabilities, // 预先静态配置；子实例首次 spawn 时与之对齐
      serverInfo: { name: 'csharp-multi-sln-router', version: '0.1.0' },
    }
  })

  // —— didOpen / didChange / didClose / didSave —— 按文件路由 ——
  for (const m of ['textDocument/didOpen', 'textDocument/didChange', 'textDocument/didClose', 'textDocument/didSave']) {
    upstream.onNotification(m, async (params: any) => {
      const uri = params.textDocument.uri
      const targets = pickInstancesByFile(uri, fileToSln, instances)
      await Promise.all(targets.map(i => i.notify(m, params)))
    })
  }

  // —— definition / hover / documentSymbol —— 单实例响应 ——
  for (const m of ['textDocument/definition', 'textDocument/hover', 'textDocument/documentSymbol', 'textDocument/signatureHelp']) {
    upstream.onRequest(m, async (params: any) => {
      const uri = params.textDocument.uri
      const targets = pickInstancesByFile(uri, fileToSln, instances)
      if (targets.length === 0) return null
      return targets[0].request(m, params)
    })
  }

  // —— references / implementation / workspace.symbol —— 广播聚合 ——
  upstream.onRequest('textDocument/references', async params => {
    const results = await broadcastActive(instances, 'textDocument/references', params)
    return dedupLocations(results.flat())
  })
  upstream.onRequest('workspace/symbol', async params => {
    const results = await broadcastActive(instances, 'workspace/symbol', params)
    return dedupSymbols(results.flat())
  })

  // —— router 自己应答的 ——
  upstream.onRequest('workspace/configuration', async params => {
    // 参考 @e:/claude-hub/claude-code/src/services/lsp/LSPServerManager.ts 现有实现
    return (params.items ?? []).map(() => null)
  })

  upstream.listen()
}
```

`scripts/router/instance.ts`（每个 Roslyn 子进程的封装，~80 行）：

```typescript
import { spawn, type ChildProcess } from 'node:child_process'
import {
  createMessageConnection,
  StreamMessageReader,
  StreamMessageWriter,
  type MessageConnection,
} from 'vscode-jsonrpc/node'
import { pathToFileURL } from 'node:url'

export class RoslynInstance {
  private proc?: ChildProcess
  private conn?: MessageConnection
  private starting?: Promise<void>

  constructor(public readonly name: string, public readonly slnPath: string, private cfg: any) {}

  isRunning() { return this.conn !== undefined }

  async ensureStarted(): Promise<void> {
    if (this.conn) return
    if (this.starting) return this.starting
    this.starting = this.spawnAndInit()
    return this.starting
  }

  private async spawnAndInit() {
    this.proc = spawn(this.cfg.roslynBin, [
      '--solutionPath', this.slnPath,
      '--logLevel', 'Information',
    ], { stdio: ['pipe', 'pipe', 'inherit'] })

    this.conn = createMessageConnection(
      new StreamMessageReader(this.proc.stdout!),
      new StreamMessageWriter(this.proc.stdin!),
    )
    this.conn.listen()

    // 与子实例完成 LSP initialize 握手
    await this.conn.sendRequest('initialize', this.cfg.initParams)
    await this.conn.sendNotification('initialized', {})
    // Roslyn LSP 启动时 --solutionPath 会自动 OpenSolution，无需再发
  }

  async request<T>(method: string, params: any): Promise<T> {
    await this.ensureStarted()
    return this.conn!.sendRequest<T>(method, params)
  }
  async notify(method: string, params: any): Promise<void> {
    await this.ensureStarted()
    await this.conn!.sendNotification(method, params)
  }

  async stop() {
    await this.conn?.dispose()
    this.proc?.kill('SIGTERM')
    this.conn = undefined
    this.proc = undefined
  }
}
```

`scripts/router/sln-parser.ts`（极简 sln 文本解析，~40 行）：

```typescript
import { readFile } from 'node:fs/promises'
import * as path from 'node:path'

const SOLUTION_FOLDER_GUID = '{2150E333-8FDC-42A3-9474-1A3956D46DE8}'

export async function parseSolutionCsprojs(slnPath: string): Promise<string[]> {
  const text = await readFile(slnPath, 'utf-8')
  const dir = path.dirname(slnPath)
  const out: string[] = []
  for (const line of text.split(/\r?\n/)) {
    if (!line.startsWith('Project(')) continue
    if (line.includes(SOLUTION_FOLDER_GUID)) continue
    const parts = line.split(',')
    if (parts.length < 2) continue
    const rel = parts[1].trim().replace(/^"|"$/g, '')
    out.push(path.resolve(dir, rel))
  }
  return out
}
```

> 这段逻辑直接对标 DotRush 的 `GetProjectsFromClassicSolution`，见 `@e:/claude-hub/claude-code/external/DotRush/src/DotRush.Common/MSBuild/MSBuildSolutionLoader.cs:36-59`

### 3.6 关键工程细节

| 问题 | 解法 |
|------|-----|
| 文件 URI → 所属 csproj | 把 csproj 的目录路径建一个**前缀 trie**；最长前缀匹配 |
| 文件不属于任何 csproj（如脚本、生成代码） | 兜底路由到 `defaultSlnName` 实例 |
| 子实例崩溃 | `ensureStarted` 里检测 conn 失活 → 标记 starting=null → 下次自动 respawn |
| 广播放大延迟 | `Promise.race` 加 timeout（如 10s），慢的子实例结果丢弃 |
| 实例闲置回收 | 每分钟检查最后活动时间戳，超时 stop |
| AI 显式指定 sln | 在 router 暴露一个 `workspace/_executeCommand` 子命令 `csharp.restrictScope("Server")` ；或经 MCP tool 注入 hint header |

## 8. 已知坑位备忘

| 坑 | 触发条件 | 规避 |
|---|---------|-----|
| Roslyn LSP `--solutionPath` 路径里有空格 | Windows 路径 | 用引号或 short path |
| 方案 A 启动期间 Claude Code 已发 `textDocument/didOpen` 但 Roslyn 还没 OpenSolution 完 | 首次启动 | `startupTimeout` 调长 + Roslyn 内部会 queue didOpen |
| 方案 B 广播 `workspace/symbol` 时部分实例还在加载 | 冷启动期 | router 给每个广播加 timeout，部分失败容忍 |
| 重叠 csproj 的 `didOpen` 被发了两次 | 方案 B | 子实例 Roslyn 内部去重，行为正确但消耗 2x bandwidth |
| AI 看到来自不同实例的同一定义重复 | 方案 B 没去重 | 在 router 层按 `(uri, range)` 去重 |
| `workspace/configuration` 不响应 | 方案 B 没实现 router 端答复 | 参照 `@e:/claude-hub/claude-code/src/services/lsp/LSPServerManager.ts` 的 handler |
